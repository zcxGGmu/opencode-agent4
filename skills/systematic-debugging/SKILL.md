# 系统化调试

## 概览

系统化调试的目标不是“让现象消失”，而是证明你理解了根因，并用最小、正确的方式修复它。

**核心原则:** 先复现，再定位，再修复，再验证。不要猜。

## 流程

### 1. 复现

- 找到能稳定触发问题的命令、测试或操作。
- 记录实际输出、退出码和相关日志。
- 如果不能复现，先缩小复现条件，不要写修复。

### 2. 建立事实

- 阅读错误栈和失败断言。
- 找到相关代码路径。
- 明确期望行为和实际行为。
- 区分“症状”、“触发条件”和“根因”。

### 3. 形成假设

每个假设都必须可测试：

```text
如果根因是 X，那么我应该能观察到 Y。
```

不要一次改多个不相关点。一次验证一个假设。

### 4. 缩小范围

使用二分、日志、断点、最小复现或 git history 缩小范围。

常用方法：

- 在输入边界处打印或断言。
- 检查数据从入口到失败点的每一步变换。
- 对比正常路径和失败路径。
- 用 `git bisect` 或专用脚本找引入点。
- 对 flaky 问题使用条件等待和重复运行。

### 5. 修复根因

修复应直接针对已证明的根因。

避免：

- 只吞掉错误。
- 增加任意 timeout。
- 放宽测试断言。
- 添加无解释的 fallback。
- 做大范围重构来掩盖问题。

### 6. 验证

- 原始复现步骤现在通过。
- 回归测试能在修复前失败、修复后通过。
- 相关测试套件通过。
- 对高风险问题，检查相邻行为没有回归。

## 调试记录模板

```markdown
## Debugging Notes

**Symptom:** 用户可见或测试可见的失败。

**Reproduction:** 精确命令或步骤。

**Expected:** 应该发生什么。

**Actual:** 实际发生什么。

**Hypotheses:**
- [ ] H1: ...
- [ ] H2: ...

**Evidence:**
- 观察 1
- 观察 2

**Root Cause:** 已证明的原因。

**Fix:** 修改了什么，为什么直接解决根因。

**Verification:** 运行了哪些命令，结果是什么。
```

## 常见场景

### Flaky tests

优先找事件和状态条件，不要加 sleep。

- 用事件驱动等待替代固定等待。
- 控制异步执行顺序。
- 检查共享状态污染。
- 重复运行以确认稳定性。

参考：`condition-based-waiting.md`、`find-polluter.sh`。

### 数据污染

- 找到写入点。
- 找到清理缺失。
- 确认测试隔离。
- 检查全局状态、缓存、环境变量和临时文件。

### 防御纵深

如果一个错误能穿过多层边界，考虑多层防护：

- 入口校验。
- 内部不变量。
- 输出校验。
- 测试覆盖。

参考：`defense-in-depth.md`。

## 危险信号

看到这些时要停下：

- “我觉得可能是……所以先改一下。”
- 没有复现就修。
- 一个 commit 改了多个无关点。
- 测试被放宽而不是 bug 被修复。
- 把 timeout 从 100ms 改到 1000ms。
- 没有记录证据。
- 只验证新路径，没验证原始症状。

## 何时升级

以下情况应向用户说明并请求方向：

- 问题无法复现，且需要外部环境或数据。
- 根因在第三方服务或不可控依赖中。
- 正确修复需要超出当前任务范围的架构改动。
- 修复会改变公共行为或兼容性。
