# 用子代理测试 Skills

## 目的

Skill 写完后，需要验证代理是否会在正确场景触发、是否能按步骤执行、是否会在不适用场景误触发。

## 测试类型

### 1. 正向触发

给出明显适用的用户请求，观察代理是否调用 skill。

示例：

```text
用户：这个测试失败了，帮我修。
期望：调用 systematic-debugging。
```

### 2. 负向触发

给出不适用请求，确认不会误用。

```text
用户：解释一下这个 README。
期望：不调用 test-driven-development。
```

### 3. 流程执行

让代理按 skill 走一遍，检查：

- 是否能理解步骤。
- 是否需要缺失上下文。
- 是否会卡在含糊指令。
- 是否能产出要求格式。

### 4. 边界场景

测试容易混淆的场景：

- 简单任务是否仍遵循必要流程。
- 用户指令与 skill 冲突时是否尊重用户。
- 工具不可用时是否使用 fallback 或升级。

## 子代理测试 Prompt 模板

```text
你正在测试一个 skill。

Skill path: [path]
Scenario: [user request]

请说明：
1. 这个 skill 是否应该触发，为什么。
2. 如果触发，按步骤执行到第一个需要外部输入的位置。
3. 指出 skill 中任何不清楚、不可执行或缺少验证的地方。
4. 给出改进建议。
```

## 评估标准

一个 skill 通过测试应满足：

- 触发条件明确。
- 步骤可执行。
- 不要求不存在的工具。
- 有停止和升级条件。
- 有验证方式。
- 输出格式清楚。

## 常见发现

- description 太泛，导致误触发。
- 第一行动不明确。
- “验证”只写了原则，没有命令或证据。
- 占位符过多，代理不知道填什么。
- 引用路径错误。
- 与其他 skill 冲突。

发现后直接修改 skill，再重复测试。
