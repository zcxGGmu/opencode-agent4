# Anthropic Skill 编写最佳实践摘要

本文是为本项目中文化后的 Superpowers skills 保留的参考摘要。文件名保留英文，便于与 upstream 对照。

## Skill 应该是什么

Skill 是可被代理调用的专业工作流。它应提供：

- 明确触发条件。
- 可执行步骤。
- 约束和停止条件。
- 验证方式。
- 必要示例和模板。

Skill 不是长篇背景资料，也不是“做得更好”的泛泛建议。

## Frontmatter

每个 `SKILL.md` 必须有 YAML frontmatter：

```yaml
---
name: my-skill
description: 在什么情况下使用，以及它完成什么。
---
```

`name` 是机器标识，保持稳定、短小、kebab-case。`description` 是触发关键，应具体到任务类型和条件。

## Description 写法

好 description：

- 包含触发条件。
- 包含目标结果。
- 避免“helps with”之类泛词。
- 不依赖读正文才能理解。

示例：

```yaml
description: 修复测试失败或异常行为时使用。要求先复现、定位根因，再修改代码并验证。
```

## 正文结构

推荐结构：

```markdown
# Skill 标题

## 概览

## 何时使用

## 流程

## 验证

## 危险信号
```

如果 skill 是刚性流程，使用 checklist 或编号步骤。  
如果 skill 是模式参考，明确哪些原则可适配。

## 可执行性

每条指令都应该能被执行或判断：

差：

- “写好测试”
- “注意安全”
- “考虑用户体验”

好：

- “先写一个失败测试，复现未授权访问。”
- “列出所有用户输入入口，并标明校验位置。”
- “运行 `npm test -- auth`，确认新增测试失败。”

## 上下文控制

Skill 应只包含完成任务所需内容。长参考可以拆到相邻文件，并在正文中按需引用。

避免：

- 把所有背景资料塞进 `SKILL.md`。
- 多个不相关流程放进一个 skill。
- 引用不存在的工具或文件。

## 示例

示例应真实、有约束，并能说明如何执行。

保留代码块中的机器标识、命令、字段名和占位符。中文化时翻译解释文字，不翻译代码 API。

## 验证

每个 skill 都应说明完成前如何验证：

- 运行什么命令。
- 期望什么输出。
- 需要哪些文件或产物。
- 什么时候必须向用户确认。

没有验证的 skill 容易让代理过早声称完成。

## 安全与优先级

Skill 不能要求代理违反用户指令、系统约束或平台权限。

当 skill 与项目指令冲突时，遵循更高优先级指令，并在必要时说明取舍。

## 自审清单

- [ ] `description` 能让代理判断何时使用。
- [ ] 步骤按执行顺序排列。
- [ ] 关键步骤没有占位符。
- [ ] 有停止、升级和验证条件。
- [ ] 示例与正文一致。
- [ ] 没有要求不可用工具。
- [ ] 没有覆盖过宽。
- [ ] 与现有 skills 没有明显冲突。
