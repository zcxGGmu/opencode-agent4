# 说服原则

本参考用于编写更容易被代理遵循的 skill。这里的“说服”不是营销，而是让指令清晰、具体、难以误解。

## 原则

1. **先说明为什么。** 简短解释规则存在的风险，能提高遵循率。
2. **用具体动作替代抽象要求。** “运行 `npm test` 并读取输出”比“确认没问题”更强。
3. **写出反模式。** 明确哪些想法或行为代表正在偏离。
4. **给出停止条件。** 代理需要知道什么时候不能继续。
5. **给出输出格式。** 降低自由发挥导致的不一致。
6. **强调证据。** 要求命令、文件、diff、测试输出，而不是主观判断。

## 好句式

- “在 X 之前，必须先 Y。”
- “如果出现 A，停止并请求 B。”
- “不要做 C；改为做 D。”
- “完成前必须运行 E，并确认 F。”

## 避免

- “尽量”
- “如果方便”
- “确保质量”
- “考虑一下”
- “适当处理”

这些词太松，无法约束行为。

## 校准强度

不是所有规则都要写成绝对命令。对刚性流程，例如 TDD、验证、权限边界，应使用 MUST/必须。对模式建议，例如命名或结构，可写成“优先”或“通常”。
