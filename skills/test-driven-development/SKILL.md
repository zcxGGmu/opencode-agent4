# 测试驱动开发

## 概览

TDD 是实现纪律，不是测试补丁。先用测试定义行为，再让实现满足行为。

**核心原则:** Red -> Green -> Refactor。没有先失败的测试，就没有可信的修复。

## 必须遵循的循环

1. **Red:** 写一个能表达目标行为的测试。
2. **Run:** 运行该测试，确认它因为预期原因失败。
3. **Green:** 写最小实现，让测试通过。
4. **Run:** 运行目标测试，确认通过。
5. **Refactor:** 在测试保护下清理实现。
6. **Run suite:** 运行相关套件，确认没有回归。

不要跳过 Red。一个从未失败过的测试不能证明它覆盖了 bug。

## Bug 修复的回归测试

修复 bug 时，测试必须复现原始症状：

```text
1. 写测试，表达 bug 应该如何表现。
2. 在修复前运行，确认失败。
3. 实现修复。
4. 运行测试，确认通过。
5. 可行时回退修复，再运行测试确认失败，恢复修复后再确认通过。
```

最后一步是强证明：测试确实能抓住 bug。

## 写测试前先定义行为

每个测试都应回答：

- 用户或调用方期望什么行为？
- 输入是什么？
- 输出或副作用是什么？
- 错误场景如何表现？
- 这个测试失败时能明确指向什么问题？

不要测试实现细节，除非它是稳定的公共契约。

## 测试粒度

优先顺序：

1. **单元测试:** 快速、聚焦，适合纯逻辑和边界条件。
2. **集成测试:** 验证模块协作、I/O、数据库、文件系统。
3. **端到端测试:** 只覆盖关键用户路径，避免过度脆弱。

测试应该少而准，覆盖真实风险，而不是追求数量。

## 常见反模式

- 先写实现，后补“看起来会过”的测试。
- 测试没有在修复前失败。
- 测试只验证 mock 调用，不验证用户可见行为。
- 为了让测试通过而削弱断言。
- 用 sleep 或任意 timeout 处理异步。
- 测试太宽泛，失败时不知道哪里坏了。
- 测试太依赖内部结构，重构就碎。

更多反模式见 `testing-anti-patterns.md`。

## 异步与时序

不要用任意等待替代条件等待。

优先：

- 等待事件。
- 等待状态变化。
- 等待 promise 或 callback。
- 使用可控 clock 或 fake timer。

避免：

```javascript
await new Promise(resolve => setTimeout(resolve, 1000));
```

更好：

```javascript
await waitFor(() => expect(store.status).toBe("ready"));
```

## 测试命名

测试名应描述行为，而不是实现：

```text
好：returns validation error when blueprint id is missing
差：test validateBlueprint()
```

中文项目可用中文描述，但保留代码标识符。

## 最小实现

Green 阶段只写让测试通过所需的最小代码。不要顺手实现未测试需求。

如果发现需要更多行为，先补测试，再实现。

## 重构

只有测试通过后才重构。重构后运行相关测试。

重构目标：

- 去重。
- 提升命名。
- 收紧边界。
- 简化控制流。
- 删除不必要抽象。

不要在 Red 或 Green 阶段做大重构。

## 验证完成

声称完成前：

- 新测试在修复前失败过。
- 新测试在修复后通过。
- 相关测试套件通过。
- 对高风险改动，运行更广范围测试。
- 失败输出与通过输出都已读过。

需要声明完成时，配合 `verification-before-completion` 使用。

## 适用与例外

适用：

- 新功能。
- bug 修复。
- 共享逻辑重构。
- 数据契约、权限、安全边界。

可能不适用：

- 纯文档修改。
- 机械重命名或格式化。
- 无可执行行为的资产更新。

即使例外，也要说明为什么不写测试，以及用什么验证替代。
